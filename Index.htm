It Doesn't Have to End in Tiers

Andrew Gunn (@andrewgunn) - Tue 02, December 2014 6:30pm

Are you fed up of working on systems crippled by "traditional" software architecture? The classic n-tier architecture seems to be the one of choice for the majority of applications today, especially the web variety, comprising of several layers such as presentation, business logic, and data. Whilst it seems like a good idea on paper, cons include leaky abstractions throughout the layers, and the reliance on a normalised database that only contains the latest state and couples every consumer to it's schema.

Command Query Responsibility Segregation (CQRS), Domain Driven Design (DDD), and Event Sourcing *could* be the silver bullet...

CQRS forces developers to separate the write/read concerns by using commands and queries respectively. By doing so, each command/query have a single responsibility, whilst giving clear contextualisation. DDD compliments CQRS by modelling the business' requirements by creating a rich domain, without giving any consideration to the storage mechanism. The building blocks include aggregates, entities, value objects, domain events, services, and repositories. Command handlers interact with the aggregate(s) to perform business operations based on a particular command. By utilising Event Sourcing, each change of state is encapsulated within a domain event, serialised, and persisted. Instead of selecting rows from a database, domain objects are hydrated by replaying each event in the same sequence.

This presentation will provide a high level overview of CQRS, DDD, Event Sourcing, and how to use them together. The best way to do this is through code, code, and more code. It'll also cover some of the classic pitfalls with a distributed architecture such as versioning. By the end, you should have a good understanding of why you'd choose such a drastic architecture, and more importantly, when not to.
